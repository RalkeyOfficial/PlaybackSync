# Extension — What's Left

Tracker for the browser-extension work. Done sections live in [`extension/docs/`](extension/docs/); this is the punch list for everything still missing.

Detailed slice plans live under [`agent-os/specs/`](agent-os/specs/) — each item below becomes its own spec when picked up.

## Already shipped

- Plugin foundation — `Adapter` contract, content runtime, `_template` adapter ([spec](agent-os/specs/2026-05-24-0959-extension-plugin-foundation/)).
- v2 WebSocket client — JOIN/EVENT/HEARTBEAT/CLOCK_PING/BUFFER + ROOM_STATE/STATE/CURSOR_CHANGE/PLAYLIST_UPDATE/SYNC_ADJUST handling, reconnect with replay, feedback-loop suppression ([spec](agent-os/specs/2026-05-24-1230-extension-ws-client/)).
- Credential pickup from the share URL — dedicated `credentials.content.ts` sniffs `?sync_url=&sync_password=` at `document_start`, background is first-write-wins on `pbsync` ([spec](agent-os/specs/2026-05-24-1423-extension-share-url-creds/)).
- First real site adapter (miruro) — scoped video selector, `MutationObserver` for late Vidstack hydration, synthesized `Space` keydown/keyup on the cold-page manual-load button, identity derived from `/watch/<showId>?ep=<ep>` ([spec](agent-os/specs/2026-05-24-1700-extension-miruro-adapter/), [doc](extension/docs/adapter-miruro.md)).
- Toolbar popup — Port-based snapshot channel, derived `PopupStatus` (no_credentials / connecting / joined / disconnected), cursor block with provider + label + clickable pageUrl, Leave Room button that wipes creds and tears down the socket. Vanilla TS, framework-free. Manual creds entry deliberately omitted — credentials only enter via the share URL ([spec](agent-os/specs/2026-05-24-2031-extension-toolbar-popup/), [doc](extension/docs/popup.md)).
- Real `nudge-rate` for `SYNC_ADJUST` — adapter contract gained `setPlaybackRate(rate)`; runtime owns the rate math + restore timer (`±5 %` clamp, `≤ 3 s` window). The `sync_adjust { delta }` command variant is gone in favor of `nudge_rate { targetPos }` ([spec](agent-os/specs/2026-05-24-1830-extension-nudge-rate/)).
- Multi-room / multi-tab arbitration — one WS runtime per syncing tab, pooled by `chrome.tabs.id`; per-tab `pbsync.tab.<tabId>` creds slots; popup retargeted to the popup-opening tab via `subscribe`; share-URL pickup binds to the capturing tab; cold-boot sentinel wipes orphan slots on browser restart ([spec](agent-os/specs/2026-05-25-1530-extension-multi-tab-arbitration/)).
- `currentlyShowing` + `catalogFragment` reporting on JOIN — optional `scrapeCatalog()` on the `Adapter` contract; per-tab `WsRuntime` defers the first JOIN up to 3 s waiting on identity + catalog from the content script; miruro scrapes the watch-page episode list via a scoped `MutationObserver` ([spec](agent-os/specs/2026-05-25-1645-extension-currently-showing-catalog/)).
- Viewer-driven `CURSOR_CHANGE_REQUEST` via episode clicks — delegated click listener on miruro's episode list emits `cursor_trigger`; background arbitrates per room mode (default + in-playlist → send; single or default-out-of-playlist → soft-leave; freeform → send unconditionally, server auto-appends not-in-playlist targets in the same transaction). Soft-leave keeps the per-tab creds slot so the popup surfaces a one-click Rejoin alongside Leave room, with a `softLeftTabs` gate that stops the status heartbeat from auto-reconnecting. Receiver side replays the cursor change by synthetically clicking the matching episode button (matched by parsed `?ep=`, not playlist order, so owners can reorder freely); `Event.isTrusted` filters the synthetic click out of the announce path. Server-side, `JoinHandler` unicasts a `PLAYLIST_UPDATE` after `ROOM_STATE` so the joiner can resolve targets against the room's known set. ROOM_STATE wire format unchanged ([spec](agent-os/specs/2026-05-25-2107-extension-owner-cursor-change/)). Freeform-click and dormant `sendPlaylistUpdate` API plumbing landed in the follow-up ([spec](agent-os/specs/2026-05-26-1500-extension-playlist-update/), [doc](extension/docs/playlist-update.md)).
- Mode suggestion from the bootstrap URL (nextcloud dashboard) — the room-creation modal classifies the pasted URL by shape (`src/util/suggestMode.ts`: bare YouTube/Vimeo → freeform; YouTube `?list=` / miruro / Crunchyroll → default/no-nudge) and surfaces a dismissable `NcNoteCard` nudge toward Freeform with one-click apply; never auto-overrides. Same change reworked the mode picker from two mutually-exclusive switches into a three-option radio group (Default/Single/Freeform, default Default) and fixed a dead-toggle bug (the switches listened for the legacy `@update:checked` event that `@nextcloud/vue` 9 no longer emits) ([spec](agent-os/specs/2026-05-30-1110-suggest-mode-from-bootstrap-url/)).
- Supervise the WS daemon (dev + production) — replaced the bare `docker exec -d` launcher with a supervised compose sidecar (`docker-compose.ws.yml`, `restart: unless-stopped` + `/healthz` healthcheck; `start-ws-server` now drives it), shipped a parameterized `docker-compose.ws.example.yml` for self-hosters, and added a graceful `SIGTERM`/`SIGINT` shutdown to `lib/Command/WsServe.php` (`$loop->stop()` → exit 0). Binding stays in admin settings; the compose `command` no longer pins `--host`/`--port` ([spec](agent-os/specs/2026-05-30-1222-supervise-ws-daemon/)).
- Restart the WS daemon from admin settings (no terminal) — Design A: HMAC-authed `POST /admin/restart` on `PresenceHttpServer` responds 200 then stops the loop so `occ` exits 0 and the supervisor starts a fresh process; `AdminRestartClient` + `AdminSettingsController::restartDaemon()` (`POST /api/v1/admin/ws/restart`, 502 on transport failure); a prominent **Daemon control → Restart daemon** button in admin settings that polls `/api/v1/ws/status` until the daemon is back (and an auto-prompt after saving binding changes). No supervisor → the readiness poll reports it didn't come back ([spec](agent-os/specs/2026-05-30-1336-ws-daemon-restart-button/)).
- `SIGHUP` config-reload — zero-downtime tunable reload: `WsConfig::reloadFrom()` (drops the per-process `IAppConfig` cache, re-reads, returns the changed-keys map) driven by a `SIGHUP` handler in `WsServe` **and** an HMAC-authed `POST /admin/reload` (`ReloadController`, `AdminReloadClient`, `AdminSettingsController::reloadDaemon()`). The admin WS-tuning / Room-defaults saves auto-apply to the running daemon (`PresenceController` now reads `maxClientsPerRoom` live). Binding keys / `ws_admin_secret` / `ws_event_log_size` stay restart-only; rate limits apply to new connections ([spec](agent-os/specs/2026-05-30-1500-sighup-config-reload/)).

## Cleanup / quality (any time)

- **Pre-submission manifest sanity pass** — `host_permissions` is now an adapter-derived allowlist sourced from [`extension/src/adapters/host-matches.ts`](extension/src/adapters/host-matches.ts) (no more `<all_urls>`); store-listing copy + privacy policy live in [`extension/docs/store-listing.md`](extension/docs/store-listing.md) and [`extension/docs/privacy.md`](extension/docs/privacy.md). What is still left before a real submission:
  - Decide whether the `tabs` permission can be dropped — the current code only uses `chrome.tabs.onRemoved`, `chrome.tabs.sendMessage`, and `chrome.tabs.query({})` for tab IDs, none of which strictly require it once `host_permissions` covers the synced sites. AMO will ask either way; the cheapest answer is "we don't use it" if a small refactor confirms that.
  - Capture the five screenshots described in [`extension/docs/store-listing.md`](extension/docs/store-listing.md) §Screenshots against a live miruro page.
  - Bump `manifest.version` from `0.1.0` when an actual submission happens; the stores reject re-submissions of the same version.
  - Stand up a Chrome Web Store + AMO developer account, and host the privacy-policy text from [`extension/docs/privacy.md`](extension/docs/privacy.md) at a public URL the listing can point to.
  - Re-verify the `_template` smoke-test path documented in [`extension/README.md`](extension/README.md) — the runtime content script no longer matches `<all_urls>`, so the bundled `template-test.html` page (loaded under `chrome-extension://<id>/`) may need either a dev-only manifest override or a direct script import from the HTML itself.

## Deferred (later phases)

- **In-flow "contribute this page" UI** — the dormant [`sendPlaylistUpdate`](extension/src/background/ws.ts) caller + the freeform-click fix shipped in [`agent-os/specs/2026-05-26-1500-extension-playlist-update/`](agent-os/specs/2026-05-26-1500-extension-playlist-update/) ([doc](extension/docs/playlist-update.md)). What's still missing is a UI: a popup "Add this page" button doesn't work because using it would mean navigating away from the room's current video, which itself triggers a soft-leave. Blocked on an adapter that can surface rich metadata (title, episode info, …) for a page outside the conventional in-page episode-list pattern and decide what "contributing this page" means in its own context — at which point that adapter brings its own port-message kind and popup affordance and calls `sendPlaylistUpdate` from there.
- **Browser-wide shared `clientId` (admin "one identity per browser" view).** Idea parked while shaping multi-tab arbitration ([`agent-os/specs/2026-05-25-1530-extension-multi-tab-arbitration/shape.md`](agent-os/specs/2026-05-25-1530-extension-multi-tab-arbitration/shape.md) §Considered & deferred). Today each tab mints its own `clientId`; admins see N rows for one human in N rooms. The backend already keys clients per-room (`RoomRuntime.$clients` in [`lib/WebSocket/RoomRuntime.php:21-48`](lib/WebSocket/RoomRuntime.php#L21-L48)), so reusing one `clientId` across **different** rooms is safe. The blocker is the same-room case: the server rejects two live sockets sharing a `clientId` in one room with `CLIENT_ID_IN_USE` ([`lib/WebSocket/Handler/JoinHandler.php:247-248`](lib/WebSocket/Handler/JoinHandler.php#L247-L248)). The natural fix is pivoting from per-tab WS to **per-room WS** (one socket multiplexed across all tabs of one room) — which revives the within-room arbitration question (heartbeat owner, command fan-out) we sidestepped by going per-tab. Pick this up when admin UX pain justifies the redesign.
- **ESLint enforcement of the documentation policy** (`jsdoc/require-description` etc.). Currently a review check.
